Software Requirements Specification (SRS)

1. Introduction

1.1 Purpose

This document describes the technical requirements for the development of a mobile application that allows users to:

  • track the menstrual cycle and symptoms,
  • store data locally and in the cloud with end-to-end encryption,
  • transfer data between devices while maintaining confidentiality.

1.2 Scope

  • Supported platforms: Android
  • Core features: local cycle tracking, history, prediction, symptom log, cloud backup, data recovery
  • Main constraints: minimal resource usage, full data privacy, no analytics or advertising

2. Overall Description

2.1 Product Perspective

The application is a standalone mobile product with a client–server architecture used for optional cloud storage of encrypted data.

Components:

  • Client (mobile application): UI, local storage, encryption
  • Server (cloud storage): storage of encrypted data without access to its contents

2.2 User Characteristics

  • Women before menopause, pregnant women
  • Value data privacy
  • May change devices and want to restore their data

2.3 Constraints

  • Use of free services / SDKs
  • Support for low-end devices
  • GDPR and Google Play requirements

3. Functional Requirements

ID Requirement Description
FR-1 Local storage Cycle, symptom, and prediction data are stored locally on the device
FR-2 Add cycle The user can add the start and end dates of a cycle
FR-3 View history Cycle history is displayed in a calendar view
FR-4 Prediction The system predicts upcoming cycles based on entered data
FR-5 Symptom log The user can log symptoms by date
FR-6 Edit / delete data The user can edit or delete any stored data
FR-7 Optional cloud backup The user can enable cloud backup
FR-8 End-to-end encryption All data is encrypted on the client before being uploaded to the cloud
FR-9 Per-record encryption Each cycle entry is encrypted individually
FR-10 Data recovery Data can be restored on a new device using the encryption key
FR-11 Data deletion The user can fully delete local and cloud data

UC-1: Start using the app without registration

Primary Actor: User

Goal: Start using the app without creating an account

Related Requirements: FR-1

  • The app is launched for the first time

  • Local storage is initialized

Primary Actor: User

Goal: Use the app with cloud backup and cross-device data recovery

Related Requirements: FR-7, FR-10

Preconditions

  • The app is installed

  • The user opens the app for the first time
  • The user taps “Sign in with Google” on the Profile screen

  1. The app starts.
  2. The user is offered Google authentication.

UC-3: Add a new menstrual cycle

Primary Actor: User

Goal: Record the start and end of a cycle

Related Requirements: FR-2, FR-6

Preconditions

  • The app is running
  • The user is on the Calendar screen

Trigger

The user taps “Log period”

Main Flow (Happy Path)

  1. The app displays an input form:
    • cycle start date
    • cycle end date
  2. The user enters data.
  3. The user confirms saving.
  4. The app stores data locally.

Postconditions

  • The new cycle is saved locally
  • The data is available for viewing and prediction

4. Non-Functional Requirements

4.1 Performance

  • UI response time ≤ 1 second
  • Data save/load time ≤ 1 second

4.2 Reliability

  • No data loss when switching devices
  • Backup does not affect app usability

4.3 Security

  • Data encryption: AES-256 or equivalent
  • Encryption key is stored only by the user
  • The server has no access to decrypted data

4.4 Usability / Accessibility

  • Intuitive interface
  • Readable fonts and high contrast
  • Minimal steps for core scenarios

5. System Architecture

5.1 Client

  • Data storage: SQLite / Room (Android)
  • Encryption: AES-256, key generated on device
  • UI: simple screens for tracking, history, symptom log, cloud

5.2 Server

  • Server type: cloud storage (Firebase)
  • Storage: encrypted data only, no decryption
  • API: CRUD operations over HTTPS
  • Registration / login: optional via local unique identifier

6. Data Models

Core entities:

  1. CycleEntry
    • startDate: Date
    • endDate: Date
    • symptoms: list[string]
    • notes: string (optional)
  2. UserData
    • encryptionKey: string (stored on client only)
    • cloudBackupId: string (technical cloud identifier)

7. Interfaces

  • Client ↔ Server: REST API
  • Client ↔ Local Storage: encrypted local storage
  • Cloud backup: encrypted JSON / binary blobs

8. Acceptance Criteria

  • The user can start using the app without registration
  • Adding, viewing, editing, and deleting data works correctly
  • Cloud backup works after being enabled and encrypted
  • Data is restored on a new device without loss
  • Data is fully deleted locally and in the cloud
  • The UI is responsive and intuitive

9. Risks & Mitigation

Risk Probability Impact Mitigation
Encryption errors Medium Very High Use proven AES-256 libraries, avoid custom crypto
User loses encryption key Medium High Clear instructions on secure key storage
Data loss due to failure Medium High Local and cloud backups, atomic saves
Resource limitations High Medium Low-end optimization, minimal MVP
