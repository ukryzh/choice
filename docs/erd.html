Entity-Relationship Documentation

The choice data model is intentionally narrow. Three entities are stored on the device; only one of them ever leaves the device, and only as ciphertext.

1. Overview Diagram

The diagram below shows entities, cardinalities, and which fields are encrypted before being uploaded to cloud backup.

2. Entities

2.1 UserData

A single per-installation record. Holds the device-side encryption key, the local access code hash, and an optional Google account identifier used only as backup identity. UserData never leaves the device.

FieldTypeNotes
userId string (PK) UUID generated locally on first launch.
encryptionKey string AES-256 key, kept in the platform secure store (Android Keystore / iOS Keychain). Never uploaded.
cloudBackupId string Opaque identifier used to address the user’s ciphertext bucket.
accessCodeHash string Salted hash of the four-digit app access code.
googleAccountId string Only present if the user opted into Google Sign-In for backup.

2.2 CycleEntry

One row per logged cycle. Encrypted per-record before upload.

FieldTypeNotes
cycleId string (PK) UUID, generated locally.
userId string (FK) References UserData.userId.
startDate Date Required.
endDate Date Optional — open cycles are allowed.
notes string Free-form, optional.
createdAt Timestamp For ordering and conflict resolution.

2.3 SymptomLog

A symptom record for a specific day within a cycle. Encrypted per-record before upload.

FieldTypeNotes
logId string (PK) UUID.
cycleId string (FK) References CycleEntry.cycleId.
date Date The day the symptoms apply to.
symptoms string[] List of tag-like values (e.g. cramps, headache, fatigue).
mood string Optional mood label.

3. Relationships

  • UserData → CycleEntry: 1 : N. Each user owns many cycles.
  • CycleEntry → SymptomLog: 1 : N. Each cycle can have many symptom entries — typically one per day.

4. Encryption Boundary

Plaintext records exist only inside the application’s sandbox on the device. Anything that crosses to Firebase is wrapped in AES-256 ciphertext keyed by UserData.encryptionKey. The cloud sees only:

  • cloudBackupId — an opaque address
  • A ciphertext blob per record
  • Metadata required for sync (record id, version)

There is no cloud-side equivalent of UserData: the credentials that make decryption possible never leave the device.

5. Out of Model

  • No analytics events.
  • No advertising identifiers.
  • No location, contacts, or device-fingerprinting fields.
