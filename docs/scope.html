Scope & Vision Document

1. Vision

1.1 Background

Women are forced to use menstrual cycle tracking applications that sell or share their personal data due to the lack of privacy-respecting alternatives.

With the rapid development of AI assistants, individual users are now able to build applications independently, making it possible to create software that does not monetize or exploit user data.

1.2 Vision Statement

For women who track their menstrual cycle, I am building a mobile application that addresses the problem of personal data protection and user control, unlike existing competitors on the market.

1.3 MVP Goals

  • Business goal: provide users with a secure and privacy-first alternative for menstrual cycle tracking
  • Success metric: a fully functional application developed independently and published on Google Play

1.4 Target Audience

  • Women before menopause
  • Non-pregnant women

1.5 Value Proposition

  • Core value: the ability to track the menstrual cycle while maintaining data security, full user control, and data portability
  • Unique Selling Proposition (USP): strict privacy and data protection by design

2. Scope

2.1 In Scope

Core Features
  • Log the start and end dates of the menstrual cycle
  • View historical cycle records
  • View predicted dates for upcoming cycles
  • Track and record symptoms for specific cycle days
  • Export and save personal data from the application
  • Create encrypted cloud backups without exposing personal data
  • End-to-end encryption for cloud backups (user-initiated)
  • Synchronize data across user devices
Platforms
  • Mobile (initially Android)

2.2 Out of Scope

Explicitly Excluded
  • Any monetization functionality
  • Server-side analysis of user data
  • Use of user data for advertising or research purposes
  • Sharing data with third parties
Post-MVP Ideas
  • Tracking additional medical conditions (e.g. ovulation)

2.3 Assumptions

  • Users value data privacy and do not want their data sold
  • Modern AI assistants enable individuals to build applications independently
  • Users are willing to trust cloud storage if strong, transparent encryption is applied
  • Users understand the value of data backups

2.4 Constraints

  • Time: no fixed deadlines
  • Budget: free-tier services only
  • Resources: personal time only
  • Regulatory: GDPR, HIPAA
  • Technical: requirement for secure server infrastructure

3. High-Level Requirements

3.1 Functional Requirements

  • Users can use the application without registration
  • No personal data is shared with third parties
  • Manual input of cycle start and end dates
  • Cycle history displayed in calendar format
  • Cycle predictions based on user input
  • Symptom tracking for specific days
  • Editing and deleting any previously entered data
  • All data stored locally on the user’s device
  • Optional encrypted cloud backup
  • Ability to restore data on a new device
  • Server has no access to decrypted user data
  • Encryption keys are stored only by the user

3.2 Non-Functional Requirements

Performance

All user actions (save, view, navigation) must complete within ≤ 1 second.

Security
  • All data belongs exclusively to the user
  • No third-party data sharing
  • Client-side encryption for all sensitive data
UX / Accessibility
  • Intuitive interface without onboarding
  • Readable fonts and sufficient contrast
Reliability
  • No data loss when switching devices
  • Backups do not interfere with core functionality

4. Success Criteria

Launch Success

  • Application published on Google Play
  • No moderation issues related to personal data handling

Quality Metrics

  • Stable application launch on supported devices
  • All core user scenarios work without errors
  • User data persists across sessions
  • No data leaks or third-party trackers

Acceptance Criteria

  • Users can start using the app without registration
  • Users can add cycles, view history, see predictions, and log symptoms
  • Users can delete all their data
  • No ads or analytics trackers are present
  • Server receives only encrypted data for backup purposes
  • Users can migrate data to a new device
  • Encrypted data remains unreadable if the server is compromised
  • Users can delete cloud backups at any time

5. Risks & Mitigation

Risk 1: Limited Resources (Single Developer)

  • Probability: High
  • Impact: Medium
  • Mitigation: Strict MVP scope, avoidance of complex algorithms

Risk 2: Cycle Prediction Accuracy

  • Probability: Medium
  • Impact: Medium
  • Mitigation: Clear disclaimers, no medical advice

Risk 3: Regulatory Constraints (GDPR, HIPAA)

  • End-to-end encryption on the client side
  • No server-side data processing
  • User control over backups and data deletion
  • Privacy by design and by default
  • Transparent and clear privacy policy

Risk 4: User Trust

  • Transparent privacy policy
  • Open explanation of data storage architecture

Risk 5: Loss of Encryption Key

  • User warnings
  • Clear backup instructions

6. Milestones & Phases

MVP

  • Basic cycle tracking
  • Local data storage
  • Basic predictions
  • Data encryption
  • Encrypted cloud backups
  • Data recovery on a new device

Beta

  • Testing with real users
  • Bug fixing
  • UX improvements

Release

  • Preparation and Google Play release
  • Final documentation and privacy policy

Post-Release Iterations

  • Improved prediction algorithms
  • Additional symptom tracking
  • Expanded functionality (Post-MVP)
