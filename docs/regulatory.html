Regulatory Compliance Analysis

This document maps the choice application against the regulatory regimes most relevant to a privacy-first menstrual cycle tracker: GDPR (EU), HIPAA (US), and Google Play health and personal data policies. The analysis is high-level and does not constitute legal advice.

1. Scope of Compliance

1.1 Personal Data Handled

  • Menstrual cycle dates (start, end)
  • Symptom logs attached to specific days
  • Cycle predictions derived from user input
  • Optional Google account identifier (for backup identity only)

No device identifiers, location, contacts, advertising IDs, or behavioural metadata are collected.

1.2 Data Sensitivity Classification

  • Menstrual data is treated as special category data under GDPR Article 9 (data concerning health).
  • It is also health data under Google Play’s health and fitness policy.
  • It is not Protected Health Information (PHI) under HIPAA, because choice is not operated by a covered entity or business associate. HIPAA nonetheless informs the security model.

2. GDPR (EU 2016/679)

2.1 Lawful Basis

  • Local-only mode: no processing by the developer occurs; GDPR obligations are minimal because no personal data leaves the device.
  • Cloud backup mode: processing is based on explicit consent (Art. 6(1)(a) and Art. 9(2)(a)). Consent is granular, opt-in, and revocable.

2.2 Principles Compliance

Principle (Art. 5)Implementation
Lawfulness, fairness, transparency Plain-language privacy story on first launch and on the website.
Purpose limitation Data is used only for cycle tracking and recovery — not analytics.
Data minimisation No metadata, no device IDs, no behavioural events collected.
Accuracy Users can edit and delete any record.
Storage limitation Data is retained until the user deletes it; cloud backups can be wiped at any time.
Integrity & confidentiality Per-record AES-256 encryption on the client; server stores ciphertext.
Accountability Architecture is privacy-by-design; documentation is public.

2.3 Data Subject Rights

  • Access & portability: all records are visible in-app and exportable.
  • Rectification: any record can be edited.
  • Erasure: users can wipe local data, cloud data, or both.
  • Withdraw consent: turning off cloud backup deletes server-side data.
  • Object to processing: not applicable (no processing beyond what the user explicitly requests).

2.4 International Transfers

Cloud backups are stored on Firebase. Where Firebase regions outside the EU/EEA are used, data is transferred only as encrypted ciphertext that the developer and the cloud provider cannot decrypt, materially reducing transfer risk.

2.5 Privacy by Design / by Default (Art. 25)

  • The default mode is local-only, no cloud, no account.
  • Cloud backup must be explicitly enabled by the user.
  • Encryption keys are generated and stored on the device.

2.6 Data Protection Impact Assessment (DPIA)

A DPIA is recommended under Art. 35 because health data is processed at scale. Given the architectural choice that the developer cannot decrypt user data, the residual risk is low. A short DPIA will be published before public release.

3. HIPAA (US)

choice is not a HIPAA-covered entity (no clinic, insurer, or business associate involvement) and is not designed to handle PHI. HIPAA is therefore out of strict scope. However, choice voluntarily aligns with the HIPAA Security Rule as a baseline for handling sensitive health information:

SafeguardImplementation
Access control Four-digit app access code; encrypted local storage.
Audit controls Local logs of data changes; no server-side telemetry.
Integrity Atomic writes; checksum on encrypted blobs.
Transmission security TLS 1.2+ for any cloud sync; data is already encrypted at the source.
Data at rest AES-256 per-record on device and in cloud.

4. Google Play Policies

4.1 User Data Policy

  • Data Safety section will be filled in declaring: no data shared, no data collected for analytics or advertising, optional cloud backup encrypted in transit and at rest.
  • Privacy policy URL will be public before release.
  • Account deletion is exposed in-app and via the Play Store requirement (in-app and web flow).

4.2 Health Apps

  • No medical claims, diagnoses, or recommendations are provided.
  • Predictions are presented as informational only.
  • The privacy policy explicitly clarifies the non-clinical nature of the app.

4.3 Sensitive Permissions

choice does not request permissions that would trigger Google Play’s sensitive-permissions review (no SMS, call log, accessibility services, or background location).

5. Other Considerations

5.1 Reproductive-Rights Risk

Cycle tracking data has been the subject of subpoena and law-enforcement requests in some jurisdictions. The architectural commitment that the developer cannot decrypt user data is the strongest available technical mitigation: a subpoena cannot compel disclosure of plaintext that does not exist on the server.

5.2 Children

The app is not directed at children under 13 (or the local equivalent). If a minor uses the app, the privacy-by-design model still applies and no targeted data collection occurs.

5.3 Open Issues

  • Confirm which Firebase regions will be used and document them in the privacy policy.
  • Publish a short, plain-language DPIA.
  • Decide on the public license model before open-sourcing the client.

6. Summary

  • GDPR: covered by privacy-by-design architecture and explicit, granular consent.
  • HIPAA: not in strict scope; voluntary alignment with the Security Rule.
  • Google Play: compliant by virtue of minimal data collection and explicit cloud opt-in; Data Safety section to be completed before submission.
